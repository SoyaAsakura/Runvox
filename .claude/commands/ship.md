# 完全自動マージ（フルコース）

実装が終わった状態から、main へ取り込むまで一気に処理する。

## 順序

1. **`/verify`** 相当の検証を実行
   - 失敗したら中止。エラー詳細を表示してユーザー判断

2. **`/pr`** 相当のコミット + push + PR 作成
   - $ARGUMENTS を PR タイトルとして使用

3. **CI 完了まで待機**
   - SUCCESS → 次へ
   - FAILURE → 中止して原因表示
   - 進行中 → sleep 30 で polling、最大 10 分

4. **`/merge`** 相当の squash マージ + クリーンアップ
   - リモートブランチ削除
   - ローカル main 同期
   - ローカルブランチ削除
   - remote prune

## 安全策

- 各ステップ後にユーザーに進捗報告（簡潔に）
- 失敗時は **必ず中断**。勝手に rebase したり force push したりしない
- マージ前に最後に「マージしていいですか？」と確認を入れる（誤爆防止）

## 使い所

- 小さな PR で、自分で確認するまでもないとき
- ドキュメント変更など影響範囲が明確なとき
- 「夜寝る前にコミット + マージしておきたい」とき

## 使わない方が良い時

- 大きな変更（レビューが必要）
- 影響範囲が広い refactor
- 初めて触る領域
- 既存テストが落ちそうな変更
